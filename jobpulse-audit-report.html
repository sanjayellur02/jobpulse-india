🔒 Critical Fix: Whitelist profile updates

app/api/user/profile/route.ts — PUT handler
// BEFORE (DANGEROUS):
const updates = await request.json();
await supabase.from('users').update({ ...updates }).eq('id', user.user.id);

// AFTER (SAFE):
const ALLOWED_FIELDS = ['full_name','gender','dob','phone','state','district','city',
  'degree','branch','college','passout_year','percentage','skills',
  'employment_status','company_name','job_role','salary_range','expected_salary','bio'];

const rawUpdates = await request.json();
const updates = Object.fromEntries(
  Object.entries(rawUpdates).filter(([k]) => ALLOWED_FIELDS.includes(k))
);
if (Object.keys(updates).length === 0) {
  return NextResponse.json({ error: 'No valid fields to update' }, { status: 400 });
}

🔒 Critical Fix: Add rate limiting middleware

middleware.ts — add rate limit with Upstash (free tier)
import { Ratelimit } from '@upstash/ratelimit';
import { Redis } from '@upstash/redis';

const ratelimit = new Ratelimit({
  redis: Redis.fromEnv(),
  limiter: Ratelimit.slidingWindow(10, '1 m'), // 10 req/min per IP
});

export async function middleware(request: NextRequest) {
  if (request.nextUrl.pathname.startsWith('/api/auth/')) {
    const ip = request.ip ?? '127.0.0.1';
    const { success } = await ratelimit.limit(ip);
    if (!success) {
      return NextResponse.json({ error: 'Too many requests' }, { status: 429 });
    }
  }
  // ... rest of middleware
}

🔴 FINAL VERDICT: NOT PRODUCTION READY

JobPulse India has a solid concept and a reasonable technology stack choice (Next.js + Supabase). The DB schema is mostly sane. However, the current build cannot go live because: (1) a real service role key was committed and must be rotated; (2) every primary page shows hardcoded mock data that cannot be distinguished from real data by users; (3) there is zero rate limiting, mass-assignment vulnerability in profile updates, and auth tokens stored insecurely. The Google OAuth flow is broken. Charts are placeholder divs. The forgot-password page doesn't exist but is linked.

Estimated time to production-ready: 3–4 weeks of focused full-stack work.
Fix order: Security (C1–C10) → Functionality (wire real data) → DB improvements → UI polish → Performance → Monitoring.
